## 一、产品概述

### 1.1 产品信息

| 项目 | 内容 |
|------|------|
| **产品名称** | NoxTTS |
| **版本** | 1.0.0（自用版） |
| **一句话定位** | 基于 CosyVoice2 模型的 Obsidian 文本转语音插件，提供顶尖中文语音质量，并深度联动笔记内容实现智能朗读。 |
| **目标用户** | 个人自用，通勤场景每日听笔记 |
| **使用频率** | 每日 |
| **笔记规模** | 单篇 3,000 - 30,000 字 |

### 1.2 产品边界（明确不做）

- ❌ 不做语音克隆
- ❌ 不做非中文朗读
- ❌ 不发布 Obsidian 社区插件市场（自用，不跑审核流程）

### 1.3 成功标准（验收条件）

1. **稳定性**：能连续稳定朗读 30 分钟不中断
2. **后台播放**：iPhone 锁屏或切换 App 后，播放不中断，控制中心可操控
3. **表格智能朗读**：遇到表格时，自动为每个单元格添加列头前缀再朗读
4. **内容过滤**：自动跳过 emoji 等无法朗读的符号

### 1.4 成本估算

CosyVoice2-0.5B 在硅基流动平台目前推理调用免费。需注册账号获取 API Key，可能存在并发或速率限制，但免费额度足以覆盖个人每日用量。

## 二、核心功能需求

### 功能 1：基础朗读能力

#### 触发方式
- 命令面板指令：`NoxTTS：朗读当前笔记`
- 命令面板指令：`NoxTTS：朗读选中文本`
- 命令面板指令：`NoxTTS：停止朗读`
- 支持设置自定义快捷键（Mac 端专用）

#### 播放控制
- 播放 / 暂停
- 快进/快退（默认 15 秒步长）
- 可拖拽进度条
- 语速调节（0.5x - 2.5x）

#### 后台播放（关键）
- **Mac 端**：Obsidian 窗口最小化时不中断
- **iPhone 端**：锁屏、切换 App 均不中断
- iPhone 控制中心 / 锁屏界面显示播放信息与操控按钮
- 响应耳机线控的播放/暂停指令

---

### 功能 2：智能文本预处理（朗读前自动处理）

#### 2.1 Emoji 过滤
- 检测到 emoji 字符时自动跳过，不送入 TTS 引擎
- 适用范围：Unicode 表情符号（😀🔥💯等）、变体选择符
- 例外：中文标点符号（如 →、①等）保留，在中文语境下有意义

#### 2.2 表格智能朗读

**原文示例：**
特征	含义
高风险	结果很重要
低风险	结果影响小

**朗读输出应为：**
> "特征，含义。特征：低风险，含义：结果影响小。"

**处理逻辑：**
1. 解析 Markdown 表格结构
2. 提取表头作为列名
3. 遍历每一行数据，拼接为"列头：单元格值"的格式
4. 行与行之间添加短停顿
5. **表头朗读时，每个词之间稍作停顿**（如 "特征，含义。" 而非 "特征含义"），避免连读

#### 2.3 其他过滤规则
- 跳过 YAML frontmatter 内容（不朗读元数据）
- 跳过代码块内容（可配置，默认跳过）
- 跳过纯链接 URL（`http://...`），但保留链接的显示文本
- 跳过图片的 `![[xxx]]` 或 `![xxx]()` 语法
- 跳过 LaTeX 公式（`$$` 和 `$` 包裹的内容）
- **`---` 分隔线**：不朗读，直接跳过
- **`-` 列表标记符**：不朗读符号本身，替换为短停顿（150ms）
- `==高亮文本==`：去掉标记，保留文本
- `**加粗**` / `*斜体*`：去掉标记符，保留文本
- `[[内部链接]]`：只读显示文本，不读路径
- `[[链接|别名]]`：只读别名部分
- `> 引用块`：在引用块前增加"引用："前缀（可配置）
- `- [ ] 待办`：读作"待办事项：xxx"
- `- [x] 已完成`：读作"已完成：xxx"
- **大标题序数停顿**：如"二、七大技能框架"，"二"后插入短停顿再继续朗读（避免"二七大技能框架"连读）

---

### 功能 3：笔记深度联动（读什么、怎么读）

#### 3.1 标签过滤朗读
- 命令：`NoxTTS：按标签朗读`
- 逻辑：用户指定标签（如 `#review`），插件扫描当前笔记，只提取包含该标签的段落进行朗读
- 段落定义：以空行分隔的文本块，或该标签所在的列表项及其子项

#### 3.2 未复习卡片朗读
- 联动对象：Obsidian 的 Spaced Repetition 插件
- 逻辑：读取 SR 插件的复习数据，识别到期未复习的卡片，仅朗读这些卡片内容
- 输出顺序：按到期时间排序，最该优先复习的先读
- 输出格式："问题：xxx，答案：xxx"

#### 3.3 按标题导航朗读
- 命令：`NoxTTS：跳转到下一标题`
- 逻辑：在长笔记中，按 Markdown 标题层级（H1-H6）作为导航节点，用户可跳到下一章节开始朗读
- 配合目录视图使用，先听目录再选章节

### 功能 4：音频嵌入笔记

合成完成后，自动将生成的 MP3 音频嵌入到当前笔记最上方。
- 嵌入格式：使用 HTML5 `<audio>` 标签或 Obsidian 原生音频嵌入语法
- 嵌入内容包含基础播放控件：前进、后退、播放/暂停、倍速调整、进度条
- 音频文件引用缓存在 `tts_cache/` 目录的路径
- 下次打开笔记时可直接播放嵌入的音频，无需重新合成
- 若嵌入的音频缓存已失效（笔记修改后重新合成），自动更新嵌入引用

## 三、技术架构

### 3.1 整体架构
Obsidian 插件层（TypeScript）
├── 文本预处理模块（emoji过滤、表格解析、内容清洗）
├── API 调用模块（硅基流动 CosyVoice2）
├── 音频缓存模块（本地存储临时音频文件）
└── 播放器模块（后台播放、控制中心集成）

外部依赖：
└── 硅基流动 API（FunAudioLLM/CosyVoice2-0.5B）

### 3.2 核心技术决策

| 决策点 | 选择 | 理由 |
|--------|------|------|
| TTS 引擎 | 硅基流动 CosyVoice2-0.5B | 中文质量第一梯队 |
| 音频处理 | **先合成完整音频，再播放** | 保障后台播放稳定性，避免流式中断 |
| 长文本策略 | 分段合成，本地拼接 | 突破单次 API 的字数限制 |
| 移动端配置 | iOS Audio Background Mode | 必须配置，否则锁屏即停 |
| API Key 管理 | Obsidian 设置面板 | 用户可视化配置 |
| 音频缓存策略 | 基于文件修改时间戳 | 笔记未修改则复用缓存，省时省钱 |

## 四、播放流程设计

### 4.1 完整播放流程
用户触发朗读
↓
读取笔记内容
↓
文本预处理（去emoji、处理表格、去代码块、去frontmatter等）
↓
长文本分段（按段落边界，每段 ≤ 3000字）
↓
逐段调用 CosyVoice2 API 合成音频
↓
本地拼接为完整 MP3 文件
↓
缓存到本地 tts_cache/ 目录
↓
调用系统播放器播放
↓
监听播放状态，更新 UI 进度
↓
播放完毕或用户停止 → 保留缓存文件


### 4.2 音频缓存策略

- 每篇笔记基于 `文件路径 + 最后修改时间戳` 生成唯一标识
- 合成后音频保存在 `tts_cache/` 文件夹
- 下次朗读同一篇笔记时，先检查缓存：
  - 笔记未修改 → 直接播放缓存
  - 笔记已修改 → 重新合成并覆盖缓存
- 缓存超过设置上限时，按"最久未使用"原则自动清理
- 提供手动清除缓存的命令

### 4.3 预合成机制

- 命令：`NoxTTS：预合成当前笔记`
- 在 Wi-Fi 环境下提前合成全部内容并缓存
- 通勤时直接播放缓存，全程零网络依赖
- 支持多篇笔记排队预合成
- 预合成时显示进度："正在预合成 3/5 篇笔记…"

## 五、界面与交互设计

### 5.1 设计原则

- **手机优先**：iPhone 操作逻辑优先，Mac 端作为增强
- **最小视觉干扰**：通勤时主要靠听，界面只是辅助
- **物理按键优先**：能用耳机线控、锁屏控制的操作，不强制打开 App
- **界面无 Emoji**：所有播放控制界面禁止使用 emoji 图标，使用纯文本或 SVG 图标代替（PRD 5.4 浮动面板中的 emoji 在实现时替换为纯文本标签）

在 Obsidian 左侧功能区（Ribbon）显示 NoxTTS 图标，点击后执行"朗读当前笔记"命令。
- 图标使用纯文本文字 `TTS` 或 SVG 音频波形图标（不使用 emoji）

## 六、异常处理与边界情况

### 6.1 网络异常处理

#### 6.1.1 场景矩阵

| 场景 | 发生时机 | 处理策略 |
|------|----------|----------|
| 朗读前检测 | 用户触发朗读命令 | 先检测网络，无网络时直接提示，不进入合成流程 |
| 合成中途断网 | 正在调用 API 合成某一段 | 当前段重试 3 次，失败后暂停流程，提示用户 |
| 已合成部分可用 | 断网时已有 N 段合成完毕 | 保留已合成音频，网络恢复后从断点继续 |
| 播放中网络无关 | 播放本地缓存的完整音频 | 不再需要网络，断网不影响播放 |

#### 6.1.2 重试策略
单段合成失败：
↓
第 1 次重试（等待 1 秒）
↓ 失败
第 2 次重试（等待 3 秒）
↓ 失败
第 3 次重试（等待 5 秒）
↓ 失败
暂停合成流程
↓
通知用户："网络不稳定，已完成 12/20 段，请检查网络后继续"
↓
用户点击"继续合成"按钮 → 从中断处继续
用户点击"取消" → 丢弃已合成的未完成音频

### 6.2 API 异常处理

#### 6.2.1 HTTP 错误码

| HTTP 状态码 | 含义 | 处理 |
|-------------|------|------|
| 400 | 请求参数错误 | 检查文本是否为空、是否超长，截断后重试 |
| 401 | API Key 无效 | 提示用户检查 Key，停止合成 |
| 429 | 速率限制 | 等待 `Retry-After` 头指定的秒数，再重试 |
| 500 | 服务器内部错误 | 等待 5 秒重试，最多 2 次 |
| 503 | 服务不可用 | 等待 10 秒重试，最多 2 次，最终失败则暂停 |

#### 6.2.2 音频异常

| 异常 | 检测方式 | 处理 |
|------|----------|------|
| 音频文件为空 | 检查返回的音频数据字节数 | 跳过该段，标记为"合成失败"，继续下一段 |
| 音频时长异常短 | 对比输入文本长度和输出音频时长 | 若时长 < 预期的 30%，标记异常并重试 |
| 音频格式不兼容 | 解析音频头 | 转码为统一格式（MP3 128kbps），或重试 |

### 6.3 长文本分段策略

#### 6.3.1 分段原则

1. **优先在自然边界切分**：段落末尾（两个换行符）、句号、问号、感叹号后
2. **硬上限**：每段不超过 3,000 字（保护 API 调用稳定性）
3. **软目标**：每段尽量在 1,500 - 2,500 字（一次合成太多容易超时）
4. **保护表格完整性**：整个表格作为一个段落，不切碎
5. **保护列表结构**：同一列表项及其子项不切分

#### 6.3.2 分段伪代码
function splitText(text):
paragraphs = text.split(/\n\n+/) // 按空行分割
segments = []
currentSegment = ""

for each paragraph:
// 如果是表格，强制整个作为一段
if paragraph is table:
if currentSegment:
segments.push(currentSegment)
currentSegment = ""
segments.push(paragraph)
continue

if len(currentSegment) + len(paragraph) > 3000:
segments.push(currentSegment)
currentSegment = paragraph
else:
currentSegment += "\n\n" + paragraph

if currentSegment:
segments.push(currentSegment)

return segments

#### 6.3.3 段落间停顿

每段末尾自动插入静默标记（200ms 停顿），让朗读听起来有自然的段落感。
规则 1：无序列表符号处理
适用范围：无序列表标记符（- * +）

处理方式：

不朗读符号本身

替换为 150ms 停顿

停顿后直接朗读列表项文本内容

示例：

原文	朗读输出
- 高风险	[停顿 150ms] "高风险"
* 重要事项	[停顿 150ms] "重要事项"
+ 备注	[停顿 150ms] "备注"
规则 2：引号闭合后的智能断句
适用范围：右引号 " 」 之后紧跟中文内容的场景

处理方式：

右引号后紧跟换行符 → 插入 200ms 停顿

右引号后紧跟中文汉字（非虚词、非标点）→ 插入 200ms 停顿

右引号后紧跟虚词（的、了、吗、呢 等）→ 不额外停顿，正常连读

示例：

原文	朗读输出
"发生了什么"对话\n你和对方	"发生了什么" [短停] "对话" [200ms] "你和对方"
"感受"对话\n任何高难度	"感受" [短停] "对话" [200ms] "任何高难度"
"事实"的理解	"事实" [短停] "的理解"（虚词，不额外停顿）
他说："好的。"我们出发	"他说：" [短停] "好的。" [200ms] "我们出发"
规则总结对照
场景	检测逻辑	停顿时长	示例
无序列表符号	- * + 开头	150ms	- 文本 → [150ms] "文本"
引号 + 换行	右引号后 \n	200ms	"对话"\n任何 → "对话" [200ms] "任何"
引号 + 中文	右引号后直接中文汉字	200ms	"对话"你 → "对话" [200ms] "你"
引号 + 虚词	右引号后是的/了/吗/呢	0ms	"对话"的时候 → 正常连读
冒号引导引用	： + "..." 闭合处	200ms	第二层："感受"对话 → "第二层：[短停]感受[短停]对话[200ms]"

### 6.4 音频播放异常处理

#### 6.4.1 播放中断场景

| 场景 | 处理 |
|------|------|
| 来电话 | 系统自动暂停播放，挂断后用户手动继续 |
| 闹钟响起 | 系统自动暂停，闹钟关闭后用户手动继续 |
| 蓝牙耳机断开 | 自动切换到扬声器，不暂停 |
| 拔出耳机 | 自动暂停（避免在地铁外放） |

#### 6.4.2 播放异常

| 异常 | 处理 |
|------|------|
| 音频文件损坏 | 跳过当前段，标记缓存失效，下次自动重新合成 |
| 播放器被其他 App 抢占 | 停止 TTS 播放，通知用户 |

### 6.5 数据边界情况

| 场景 | 处理 |
|------|------|
| 空笔记 / 无内容 | 提示："当前笔记过滤后无可朗读内容"，不做 API 调用 |
| 纯代码笔记（过滤后 < 10%） | 询问："当前笔记大部分为代码块，朗读内容较少，是否继续？" |
| 超大笔记（> 30,000 字） | 提示预计时长，提供"朗读全文"或"选择范围"选项 |

### 6.6 存储异常处理

| 场景 | 处理 |
|------|------|
| 缓存目录不存在 | 自动创建 |
| 缓存目录不可写 | 降级到系统临时目录，提示用户检查权限 |
| 缓存超过上限 | 按"最久未使用"原则自动删除旧缓存 |
| iCloud 同步冲突 | 缓存目录排除在 iCloud 同步之外（设 `nosync` 标记） |
